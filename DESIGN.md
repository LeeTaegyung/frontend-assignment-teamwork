- **데이터 분석**: metadata.json을 어떻게 해석했나요? 데이터 구조를 어떻게 이해했나요? 더 나은 데이터 표현 방법이 있을까요?
  - 주요 필드 설명을 기반으로 각 데이터마다 연관성을 파악했습니다.
  - 데이터의 계층이 일관적이지 않아 원하는 데이터를 가져오기 위해 탐색을 깊게 해야한다는 문제점을 발견하게 되었습니다.
    - 처음엔 데이터를 바로 탐색할 수 있도록 데이터를 평탄화를 하려고 했지만 기준이 명확하지 않아 다른 방법을 강구하였습니다.
    - 전체 데이터를 확인해보니 총 3가지 패턴으로 구조가 나눠진 것을 파악하고 데이터를 재가공하여, 유틸함수를 통해 원하는 값을 가져올 수 있도록 설정하였습니다.
  - metadata.json의 데이터들이 원래는 API 형태로 봤을 때 현재 프로젝트의 정보 가져오기, 현재 도면의 정보 가져오기, 현재 리비전의 정보 가져오기 등등으로 나뉘는 것이 맞지 않을까라고 생각했습니다.

- **접근 방식**: 전체 과제를 어떤 순서와 방식으로 해결하셨나요?
  1. 시나리오를 바탕으로 기본 요구사항과 데이터를 파악했습니다.
  2. 어느 컴포넌트에 어떤 데이터가 필요한지 대략적으로 설계를 하였습니다.
  3. 페이지마다 레이아웃을 잡고 퍼블리싱 완료후 컴포넌트를 분리하였습니다.
  4. 사용자가 더 사용하기 편리한 레이아웃이나 넣으면 좋을 것 같은 부가 기능을 추가했습니다.
  5. 도면 페이지에서 메뉴를 통한 라우팅 변경 시 상태 초기화 문제를 해결하기 위해 도면 페이지의 최상단에 key를 적용했습니다.
  6. 이후 에러 처리를 추가해 안정성을 확보했습니다.

- **UI 설계 결정**:
  - 어떤 레이아웃을 선택했고, 왜 그 방식이 최선이라고 판단했나요?
    - 사용자 관점을 가장 먼저 생각했습니다. 시나리오의 김철수 소장님은 현장에서 근무하고, 도면을 확인하는 일을 하니까 태블릿으로 업무를 볼 가능성이 가장 높다고 생각했습니다.
    - 그래서 피그마나 포토샵과 같은 프로그램들의 레이아웃이 가장 적합하다 생각하여서 비슷한 레이아웃으로 가기로 하였습니다.
    - 도면 영역이 지나치게 좌우 패널에 가려지는 것은 원하지 않아서,
      - 오른쪽 사이드바 : 각 공정과 리비전을 선택할 수 있는 컨트롤 박스를 넣고 상단에는 컨텍스트의 정보를 추가했습니다.
      - 왼쪽 하단 컨트롤 버튼들 : 추가된 도면 레이어의 opacity 관리 레이어 팝업과 확대/축소 버튼을 추가했습니다.
  - 고려했던 대안들과 각각의 장단점은 무엇이었나요?
    - 다른 레이아웃도 고민해 보았지만, 현재 구조가 가장 직관적이라고 판단하여 이대로 진행하였습니다.

- **기술 선택**: 상태 관리, 스타일링 등을 어떤 기준으로 선택했나요?
  - 상태관리
    - 이번 과제에서는 전역 상태 관리 라이브러리는 사용하지 않았습니다. 주어진 데이터가 정적 데이터이고, 한번 가공을 하고나면 상태를 변경하지도 않기 때문에 전역 상태 관리는 사용하지 않았습니다.
    - props drilling이 5단계 이상이 되면 contextAPI를 고려하려고 했지만, 그 단계까지는 발생하지 않아 사용하지 않았습니다.
  - 스타일링(테일윈드)
    - 과제를 진행할 기간이 약 7일 정도였지만, 명절이다보니 실제로 쏟을 수 있는 시간은 얼마 되지 않았습니다.
    - 그래서 가장 많이 사용하기도 했고, 매번 변수를 고민할 필요 없이 사용 가능한 테일윈드를 선택했습니다.
    - Tailwind CSS는 동적인 값을 직접 지정하기 어려워 opacity 적용 시 제약이 있을 것으로 우려했지만, style 프로퍼티를 이용하면 동적인 값도 적용할 수 있어서 스타일링에 큰 제약 없이 빠르게 작업을 진행할 수 있었습니다.

- **어려웠던 점 및 개선 방안**: 과제를 수행하며 겪었던 어려움은 무엇이었고, 시간이 더 주어진다면 어떤 부분을 개선하고 싶으신가요?
  - 과제를 수행하며 겪었던 어려움은,
    - metadata.json의 데이터가 일관성이 없어서 처음에 어떻게 접근을 해야할지 고민을 좀 많이 했습니다. 결론적으로 API를 받는다면 어떻게 받는 게 좋을까를 생각해서 해결은 했지만, 시간을 생각보다 많이 소요되었습니다.
    - 도면페이지에서 처음에 컴포넌트 책임 분리를 명확하게 하지 않아 하나의 컴포넌트가 많은 일을 해야 했던 경험이 있습니다.
  - 시간이 더 주어진다면,
    - zoom 기능과 pan 기능을 추가하고 싶습니다.
    - DrawingCanvas컴포넌트에서 opacity의 상태를 담당하고 있는 layerOpacities가 props로 전달받는 selectDisciplines의 값을 기준으로 생성되었습니다. 그래서 selectDisciplines가 변경될 때 useEffect로 동기화를 해야 했습니다. LayerControlPanel 컴포넌트에서 변경하는 opacity 값을 DrawingLayer 컴포넌트로 전달해 주어야 했기 때문에 상태로 관리를 할 수 밖에 없었습니다. 성능 저하나 무한 루프 발생 가능성 측면에서 완전히 안전하다고 보기는 어렵다고 생각했습니다. useEffect 내부에서 동기화를 하지 않고 다른 더 좋은 방법이 없는지 고민하고 개선해보고 싶습니다.
